# SUUMO アナライザー — 技術選定・設計判断の記録

## なぜこのアプリを作ったか

賃貸物件を探すとき、SUUMO に掲載されている情報は「事実の羅列」に過ぎない。
築年数・賃料・間取りは目に入るが、「管理費の記載がない = 共用部の管理が個人任せかもしれない」「礼金なし + 分譲賃貸 = 短期売却リスクがある」といった **掲載情報の空白・組み合わせから読み取れる非自明なリスク** は、不動産の経験がないと見逃しやすい。

このアプリは SUUMO の物件 URL を貼るだけで、ローカル LLM がその種の非自明なリスクと具体的な確認事項を日本語で出力する。

ポートフォリオ作品としての位置づけ：スクレイピング × LLM 統合 × Next.js App Router を組み合わせたフルスタック Web アプリとして、実際に使えるツールを完成形で示すことを目的とした。

---

## アーキテクチャ

```
ブラウザ
  └─ POST /api/analyze (Next.js API Route)
       ├─ scraper.ts  → fetch → cheerio で HTML パース → PropertyData
       └─ analyzer.ts → Ollama REST API (qwen3:8b) → AnalysisResult
```

フロントエンド（`page.tsx`）・API ルート・スクレイパー・アナライザーを 1 つの Next.js プロジェクトに集約した。別サーバーを立てず、`/api/analyze` Route Handler が orchestration レイヤーを担う。

---

## 技術選定と理由

### フレームワーク: Next.js (App Router)

- API Route と UI を同一プロジェクト・同一デプロイ単位に収められる。Flask + React のような 2 サーバー構成より構成がシンプル。
- App Router は 2024 年時点での Next.js の推奨パス。
- 代替案: Remix、SvelteKit も同様の one-repo 構成が可能だが、知名度・エコシステムで Next.js を選択。

### スクレイピング: cheerio（サーバーサイド静的 HTML パース）

- SUUMO の物件スペック（賃料・間取り・築年数・費用テーブル等）は SSR で静的 HTML に含まれており、JS 実行なしで取得できる。
- cheerio は Node.js で動く jQuery 互換の HTML パーサー。ブラウザ起動不要で軽量・高速。
- 代替案として **Playwright** も検討した。周辺施設（スーパー・学校）等の情報は JS 遅延ロードのため cheerio では取れないが、物件スペック取得の目的には不要と判断してスコープ外とした。将来的に地図データが必要になれば Playwright に切り替える。

### LLM: Ollama + qwen3:8b（ローカル実行）

- **API 課金なし**・**オフライン動作**・**入力データが外部に送信されない** の 3 点が主な理由。
- ポートフォリオとして「誰でも手元で動かせる」ことを重視した（OpenAI API キー取得のハードルが不要）。
- qwen3:8b は日本語指示追従・JSON 出力安定性のバランスが許容範囲で、8GB VRAM（RTX 4060 Ti）に収まる最大サイズ。
- 既知の制約: `think: false` を指定しても `<think>...</think>` タグを混入することがあるため、レスポンス後に正規表現で除去している。
- 代替案: GPT-4o / Claude API は品質が高いが課金・プライバシー・ポータビリティの点で今回は不採用。

### プロンプト戦略: few-shot + 禁止例

- zero-shot では「築年数が古い」「初期費用が高額」など掲載情報の言い換えしか出なかった。
- **禁止出力の例**（悪い例）と**良い出力の例**を JSON ごと提示する few-shot 方式に切り替えたことで、「空白・組み合わせ由来の非自明リスク」を引き出しやすくなった。
- 制約: qwen3:8b は few-shot の論理を誤適用することがある（例: 「礼金高額 → 立退きリスク」という誤推論）。精度向上には上位モデルへの切り替えが最も効果的だが、今回はローカル動作を優先して許容範囲とした。

## スクレイパーの設計判断

### SUUMO 固有テーブルセレクターを使う理由

`$('table tr')` で全テーブルを走査すると、関連物件リストや検索フィルターのテーブルも拾ってしまい、「間取り: 駅徒歩3分以内」のような誤フィールドが生まれた（実際に発生した問題）。

SUUMO の物件詳細ページには `table.property_view_table`（主要スペック）と `table.data_table`（建物詳細・費用・契約条件）という安定したクラス名が存在するため、これを優先して使う。クラスが存在しない場合（ユニットテスト用モック HTML 等）は全テーブルにフォールバックする。

### `ul li` 抽出を廃止した理由

リンクなし `<li>` を設備情報として取得する実装を試みたが、SUUMO の実ページでは検索条件フィルター（「駅徒歩3分以内」「バス・トイレ別」）が `<li>` に含まれており、物件固有の設備と区別できなかった。テーブルと dl/dt/dd から取得できる情報で十分と判断して削除。

### 先勝ち Map による重複排除

同一ラベル（例: 「所在地」）が複数テーブルに存在する場合、先に登場した値を採用する。SUUMO の物件詳細テーブルは概要 → 詳細の順に並ぶため、より信頼性の高い値が先に来ることが多い。
